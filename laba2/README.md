# Лабораторная работа 2 — Электронно-цифровая подпись (ЭЦП)

Демонстрационное клиент–серверное приложение, показывающее практическое применение **электронной цифровой подписи** в двух сценариях: подпись на стороне клиента и подпись на стороне сервера.

## Назначение ЭЦП

ЭЦП позволяет гарантировать:

- сообщение отправлено конкретным отправителем;
- содержимое не изменялось при передаче;
- отправитель не может отказаться от факта отправки (неотказуемость).

В проекте используется **RSA** с хешированием **SHA-256** и padding **PKCS#1 v1.5**.

## Структура проекта

```
laba2/
├── app/
│   ├── __init__.py
│   ├── client.py      # CLI-клиент для обоих сценариев
│   ├── crypto.py      # KeyPair, Verifier (RSA + SHA-256)
│   ├── models.py      # Pydantic-модели для API
│   └── server.py      # FastAPI-сервер с эндпоинтами ЭЦП
├── pyproject.toml
├── uv.lock
└── README.md
```

| Файл | Описание |
|------|----------|
| `crypto.py` | Генерация RSA-ключей, подпись сообщения, верификация по открытому ключу (PEM). |
| `models.py` | Модели запросов/ответов: `VerifyRequest`, `VerifyResponse`, `PublicKeyResponse`, `SignedMessageResponse`. |
| `server.py` | HTTP-API: верификация подписи клиента, выдача открытого ключа сервера, выдача подписанного сообщения. |
| `client.py` | Запуск сценария 1 или 2 через аргументы командной строки. |

## Установка и запуск

### Окружение (рекомендуется uv)

```bash
cd laba2
uv sync
```

Либо установка в текущее окружение:

```bash
pip install -e .
```

### Запуск сервера

В одном терминале:

```bash
uv run laba2-server
```

Сервер слушает `http://0.0.0.0:8000`. Документация API: http://127.0.0.1:8000/docs .

Альтернатива — напрямую модулем:

```bash
uv run python -m app.server
```

### Запуск клиента

В другом терминале (при запущенном сервере).

**Сценарий 1** — подпись на стороне клиента (клиент вводит сообщение, подписывает, сервер проверяет):

```bash
uv run laba2-client --scenario 1
```

По запросу введите произвольное сообщение. Клиент подпишет его своим закрытым ключом, отправит на сервер для верификации и выведет результат.

**Сценарий 2** — подпись на стороне сервера (сервер генерирует и подписывает сообщение, клиент проверяет):

```bash
uv run laba2-client --scenario 2
```

Клиент запросит открытый ключ сервера и подписанное сообщение, проверит подпись локально и выведет, подлинное ли сообщение.

Указание другого адреса сервера:

```bash
uv run laba2-client --scenario 1 --url http://localhost:8000
```

## API сервера

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/verify` | Сценарий 1: тело — `message`, `signature` (base64), `public_key` (PEM). Ответ: `{ "verified": true/false }`. |
| `GET` | `/public-key` | Сценарий 2: ответ — открытый ключ сервера в PEM. |
| `POST` | `/signed-message` | Сценарий 2: ответ — случайное сообщение и подпись в base64. |
